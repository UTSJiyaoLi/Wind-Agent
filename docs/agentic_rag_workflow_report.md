# Wind-Agent Agentic RAG 工作流程与特色功能说明

## 一、定位说明

Wind-Agent 的 RAG 部分不是传统意义上“向量检索 + 大模型生成”的静态链路，而是面向风电垂域知识问答场景构建的一套 Agentic RAG 系统。它的重点不是单次召回，而是通过路由、检索质量判断、自动重试、问题拆解、答案评分和结构化输出，尽量提升风电专业问答的稳定性、可解释性和业务可用性。

从现有仓库实现看，这套 Agentic RAG 已经具备“知识库构建流程 + 在线检索控制流程 + 答案质量反馈流程”三部分能力。

## 二、整体工作流程

## 1. 离线知识库构建流程

### 第一步：文档解析

系统通过 `scripts/parse/parse_mineru_v2.py` 及相关脚本对 PDF 文档进行解析，核心目标是把原始风电文档转成结构化文本片段和可追踪的 metadata。解析后保留的信息不仅包括文本，还包括页码、图表、图片、表格等上下文线索，为后续多模态引用和检索增强打基础。

这一阶段的主要价值是：

- 将原始 PDF 文档转成可切分、可检索、可回溯的文本对象。
- 为后续图表、表格、图片等引用保留来源信息。
- 让知识库不止能回答“文本问题”，还能辅助图表和公式相关问答。

### 第二步：切分与元数据组织

解析后的文档会进一步切分为适合检索的 chunk，并写成 LangChain 风格 JSONL。系统同时将 metadata 做了分层处理：

- 轻量 metadata：用于在线检索和 Milvus 索引。
- 完整 metadata：用于精细引用、图片表格回溯和后续扩展。

这种设计的好处是在线检索性能和信息完整性兼顾，不需要把所有重 metadata 都直接压进向量库。

### 第三步：向量化与 Milvus 入库

`scripts/ingest/ingest_winddata_milvus.py` 负责将文档写入 Milvus。当前入库链路的特点是：

- 使用 BGE-M3 生成 dense 向量。
- 同时生成 BGE sparse 表征。
- 在 Milvus 内启用 BM25 稀疏检索字段。
- 同时维护 `dense_vector / bge_sparse_vector / bm25_sparse_vector` 三类检索基础。

这意味着系统天然支持混合检索，而不是单一路径召回。

### 第四步：离线评测

仓库中已经提供 Recall、MRR 与 RAGAS 评测脚本，用于验证检索质量与参数效果：

- `scripts/eval/evaluate_recall_quality.py`
- `scripts/eval/ragas_retrieval_eval.py`
- `scripts/eval/offline_regression_metrics.py`

这部分很重要，因为它说明 RAG 调优不是拍脑袋，而是已经进入可评测、可回归、可比较的工程化阶段。

## 2. 在线请求处理流程

### 第一步：模式路由

用户请求进入统一聊天服务后，系统会先做模式判断。对于 `auto` 模式，不会无条件进入 RAG，而是先判断：

- 这是不是风电垂域知识问题。
- 这是不是更适合走工具执行的任务。
- 这是不是普通通用问答，应该直接走 `llm_direct`。

因此，Agentic RAG 的第一层控制不是“怎么检索”，而是“这次是否应该检索”。

### 第二步：检索前增强

当请求进入 RAG 主链路后，系统会在检索前先做若干增强动作：

- query rewrite
- domain expansion
- top_k 等检索参数配置
- 上下文预算控制

这些动作的目的，是让风电领域中的专业术语、同义表达和复杂问法获得更高召回率。

### 第三步：混合检索

在线检索能力建立在多路召回之上。结合仓库中的运行时配置和评测脚本，当前系统的核心思路是：

- dense 检索负责语义相似召回；
- BGE sparse 检索负责词项级补充召回；
- BM25 负责关键词型、术语型和长尾表达的显式匹配；
- 多路结果再做融合。

这类混合检索特别适合风电垂域场景，因为风电文档中既有术语密集型内容，也有标准、公式、设备名词和工程表达，单一路径很容易漏召回。

### 第四步：融合、去重与重排

召回后的候选不会直接送给大模型，而是会继续做质量整理：

- 使用 RRF 对多路候选做融合。
- 按文档维度做去重和保留。
- 对候选集进行 rerank。
- 控制最终进入提示词的上下文规模。

这一步的目标是减少噪声、提升前排命中质量，让最终上下文更“有用”而不是单纯“更多”。

## 3. Agentic 控制流程

Wind-Agent RAG 的关键特点，在于加入了“检索质量反馈控制”。

### 第一步：检索质量评分

系统会根据检索结果计算一组质量信号，例如：

- top hit score
- coverage estimate
- score gap
- context count

然后综合生成 retrieval score 与 confidence，用来判断当前检索结果是否足够支撑回答。

### 第二步：自动重试

如果当前检索结果质量不够，系统不会立刻生成答案，而是自动重试。重试动作包括：

- 增大 `top_k`
- 增加 query rewrite 变体数
- 启用或强化 domain expansion

这使得系统具备“先尝试提升检索质量，再进行回答”的能力，是 Agentic RAG 相比普通 RAG 的核心差异。

### 第三步：复杂问题拆解

如果系统判断用户问题是复合问题，会先尝试拆成多个子问题，再分别检索和回答，最后统一综合。当前支持：

- 规则拆解
- LLM 辅助拆解
- 子问题并行执行
- 多子结果汇总生成

在风电场景中，这对“分别说明风速分布建模、尾流评估方法，并给出站址建议”这种复合型专业问题尤其有效。

## 4. 答案生成与结果组织流程

### 第一步：基于上下文生成回答

系统将最终选定的上下文、图片表格线索和用户问题一起交给大模型，要求答案显式基于上下文生成，并使用 `CTX` 风格引用标记关键结论。

### 第二步：答案质量评分

答案生成后，系统还会对结果继续打分，主要关注：

- grounding
- usefulness
- confidence
- hallucination risk

如果规则评分信心不足，还支持进一步调用 LLM 评分。

### 第三步：引用与预览过滤

系统不会把全部候选来源都原样展示，而是会根据最终答案中实际使用到的 `CTX` 标记，筛选真正相关的 citations 和 preview images。这样前端看到的引用列表与图片预览更贴近最终答案本身，减少无关内容干扰。

### 第四步：结构化 UI 输出

最终结果会被组装为 `ui_blocks`，前端可以按块展示：

- assistant message
- retrieval metrics
- agentic trace
- agentic grades
- subquestions
- preview images
- JSON analysis

因此，Wind-Agent 的 RAG 输出不是一段纯文本，而是一套可视化、可解释、可调试的结果结构。

## 三、特色功能总结

## 1. 面向风电垂域的混合检索体系

系统同时使用 dense、BGE sparse 和 BM25 三类检索基础，更适合风电标准、术语、设备名词、场址分析和工程文档这类兼具语义性与关键词性的知识场景。

## 2. 具备检索质量自反馈能力

系统不是一次检索失败就结束，而是会根据检索评分自动触发重试策略。这种“先纠偏检索，再给答案”的机制，是 Agentic RAG 的核心价值。

## 3. 支持复杂问题拆解

面对复合型风电专业问题，系统可以拆成多个子问题分别检索，再统一综合，减少单轮检索覆盖不足带来的答非所问问题。

## 4. 支持答案级 grounding 评估

系统不仅评估检索结果，还评估最终答案本身是否扎根于上下文，能更好控制垂域专业问答中的幻觉风险。

## 5. 支持图表、图片、表格线索回传

RAG 输出不仅有文本 citations，还支持 media refs 和 preview images，这使系统具备进一步向“风电图文知识助手”演进的基础。

## 6. 可观测、可调参、可评测

系统已经具备：

- trace/span/event 级别观测能力；
- 在线参数可调；
- 离线 Recall、MRR、RAGAS 评测脚本；
- 前端显示检索指标和 Agentic 轨迹。

这说明该系统已经不是黑盒问答，而是一个可分析、可优化、可迭代的 RAG 工程。

## 四、对风电业务的实际意义

对于风电场景，Agentic RAG 的价值主要体现在：

- 提升风电专业问题的回答准确率与可追溯性。
- 降低风电领域复杂问题的一次性检索失败风险。
- 让用户不仅获得答案，还能看到依据、轨迹和关联图像。
- 为后续接入更多风电专业工具打下统一调度基础。

从系统演进角度看，Wind-Agent 的 Agentic RAG 已经不是单独的检索模块，而是整个风电垂域智能体系统中的知识引擎和决策支撑底座。

## 五、现阶段成熟度判断

结合代码、评测脚本和前端呈现方式，当前 RAG 部分已经具备以下成熟特征：

- 已有完整的离线知识库构建链路。
- 已有混合检索与 rerank 工程方案。
- 已有 agentic 重试与问题拆解机制。
- 已有答案级评分和引用过滤机制。
- 已有前端联调和可视化调试能力。

当前仍建议继续推进的方向包括：

- 进一步统一在线检索实现与仓库内文档说明。
- 继续压实评测集与回归标准。
- 补充更多风电场景的检索样本与对抗样本。
- 对多模态引用和图片表格 grounding 做更细粒度优化。

## 六、汇报可直接使用的一段话

Wind-Agent 的 RAG 部分已经从传统静态检索增强，升级为面向风电垂域的 Agentic RAG 系统。系统不仅具备文档解析、向量化入库、混合检索、重排和引用回答能力，还引入了检索质量评分、自动重试、复杂问题拆解、答案 grounding 评估和结构化结果输出机制。它本质上已经成为整个风电垂域智能体的知识引擎，能够更稳定地支撑风电专业问答、知识检索和复杂分析任务。
